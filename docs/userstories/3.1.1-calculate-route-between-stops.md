# 3.1.1 — Calculate Route Between Consecutive Stops

**Epic:** Route Calculation
**Feature:** Leg Calculation

## User Story

As a user, I want the app to calculate the route between consecutive stops when I finalise my plan so that I know the distance and time for each leg.

## Description

When the user has set at least a starting point and destination, the app calls the Google Directions API to calculate the route. The API returns distance, duration, and route geometry for each leg (pair of consecutive stops). Results are stored locally so the user can view them without re-fetching.

## Acceptance Criteria

- Route calculation is triggered when the user explicitly requests it (e.g., a "Calculate route" button) after setting at least two stops.
- The app sends a single Directions API request with all stops (origin, destination, and intermediate stops as waypoints).
- Distance and duration are stored for each leg.
- If the API call fails (network error, invalid stops), the user is informed with a clear error message.
- Previously stored legs are replaced when the route is recalculated.

## Technical Notes

- **API:** Google Directions API. Send origin, destination, and up to 23 intermediate waypoints in a single request. The response contains a `legs[]` array with one entry per consecutive stop pair.
- **Trigger:** Calculation happens on plan creation/edit (not on-demand every time the user views the trip). Store results in the local database.
- **Data model:** `Leg(id, tripId, fromStopId, toStopId, distanceMeters, durationSeconds, polyline)` — store the encoded polyline too, even if not rendered in Level 1, to avoid re-fetching later if a map view is added.
- **Invalidation:** Legs are invalidated when stops are added (1.2.3), removed (1.2.6), reordered (1.2.4), or edited (1.2.5). Show a visual indicator that the route is stale and needs recalculation.
- **Level 2 consideration:** In Level 2, the same API is called with the user's live GPS coordinates as origin instead of the last departed stop's coordinates. Design the API client so the origin parameter is injectable.
