# 6.1.1 — GPS-Based Live Distance/Time to Next Stop

**Epic:** Live Position *(future — Level 2)*
**Feature:** GPS-based ETA

## User Story

As a user, I want the app to use my GPS to update time/distance to the next stop in real time so that I always know how far away I am.

## Description

While the user is driving, the app reads the phone's GPS position and periodically queries the Directions API from the current position to the next upcoming stop. The trip overview updates with live distance and duration values instead of the static pre-calculated ones.

## Acceptance Criteria

- The app requests location permissions from the user.
- While a trip is active and the user is driving, the app reads GPS position at a reasonable interval (e.g., every 60 seconds).
- Distance and duration to the next stop update in real time on the trip overview.
- The live values are visually distinguished from pre-calculated values (e.g., a "live" badge).
- When GPS is unavailable (no signal, permission denied), the app falls back to static values gracefully.
- Battery usage is reasonable — no continuous high-accuracy GPS polling.

## Technical Notes

- **Android API:** `FusedLocationProviderClient` for battery-efficient location updates.
- **Architecture:** Introduce a `PositionProvider` interface. Level 1 returns `lastDepartedStop.latLng`. Level 2 returns `gps.latLng`. The rest of the app (ViewModel, API client) consumes this interface without knowing the source.
- **API cost:** Each GPS update triggers a Directions API call. At one call per minute over a 5-hour drive, that's ~300 calls per trip. Well within free tier, but worth monitoring.
- **Debouncing:** Don't re-query if the user hasn't moved significantly (e.g., stuck in traffic, parked). Use a minimum displacement threshold (e.g., 500m) before triggering a new API call.
