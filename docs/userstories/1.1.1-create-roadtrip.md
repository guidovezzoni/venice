# 1.1.1 — Create a New Roadtrip

**Epic:** Trip Management
**Feature:** Trip CRUD

## User Story

As a user, I want to create a new roadtrip so that I can start planning my journey.

## Description

The user can create a new roadtrip by providing a name. The trip is persisted locally on the device. A newly created trip starts with no stops and no calculated route.

## Acceptance Criteria

- The user can enter a trip name and confirm creation.
- The trip is saved to local storage and appears in the trip list.
- A newly created trip has no stops, no legs, and zero total distance/duration.
- The user is navigated to the trip detail/editing screen after creation.

## Technical Notes

- **Data model:** `Trip(id, name, createdAt, updatedAt)`
- **Persistence:** Local database (Room recommended for Android/Compose).
- **Architecture:** Trip data should live in a Repository accessible by both the phone UI (Compose) and, in the future, Android Auto (Car App Library). Keep creation logic in a use case / ViewModel, not in the UI layer.

### User Story

### Description

From the **Trip List screen**, the user taps a **Floating Action Button (FAB)**. A **modal bottom sheet** (or `AlertDialog`) appears with a single text field for the trip name. After entering a valid name and confirming, the trip is persisted to the local Room database and the user is taken directly to the **Trip Detail screen** for the newly created trip.

A newly created trip contains no stops, no legs, and zero total distance/duration.

---

### Acceptance Criteria

- The FAB on the Trip List screen is visible and tappable.
- Tapping the FAB opens a bottom sheet / dialog with:
  - A labelled text input field (`Trip name`).
  - A **Confirm** button (disabled while the name is blank or only whitespace).
  - A **Cancel** button (or dismiss by tapping outside).
- **Validation:**
  - Name is required (cannot be empty or only whitespace).
  - Name is trimmed before saving.
  - Maximum length: **100 characters**; the UI enforces this via `maxLength` on the `TextField`.
- On successful creation the trip is persisted to Room and the user is navigated to the Trip Detail screen.
- The new trip is immediately visible in the Trip List on back-navigation.
- A newly created trip has:
  - `stops = []`
  - `legs = []`
  - `totalDistanceMetres = 0`
  - `totalDurationSeconds = 0`
- If the Room insert fails, an error `Snackbar` is shown on the Trip List screen; the bottom sheet remains open so the user can retry.

### Key Implementation Details

#### DAO

```kotlin
@Dao
interface TripDao {
    @Insert
    suspend fun insert(trip: TripEntity): Long

    @Query("SELECT * FROM trips ORDER BY createdAt DESC")
    fun observeAll(): Flow<List<TripEntity>>

    @Query("SELECT * FROM trips WHERE id = :id")
    suspend fun getById(id: Long): TripEntity?
}
```

#### Repository interface

```kotlin
interface TripRepository {
    suspend fun createTrip(name: String): Result<Trip>
    fun observeTrips(): Flow<List<Trip>>
}
```

#### Repository implementation

```kotlin
class TripRepositoryImpl @Inject constructor(
    private val dao: TripDao
) : TripRepository {

    override suspend fun createTrip(name: String): Result<Trip> = runCatching {
        val now = System.currentTimeMillis()
        val entity = TripEntity(name = name.trim(), createdAt = now, updatedAt = now)
        val id = dao.insert(entity)
        entity.copy(id = id).toDomain()
    }

    override fun observeTrips(): Flow<List<Trip>> =
        dao.observeAll().map { list -> list.map { it.toDomain() } }
}
```

#### Use case

```kotlin
class CreateTripUseCase @Inject constructor(
    private val repository: TripRepository
) {
    suspend operator fun invoke(name: String): Result<Trip> {
        if (name.isBlank()) return Result.failure(IllegalArgumentException("Name cannot be blank"))
        return repository.createTrip(name)
    }
}
```

#### ViewModel (relevant excerpt)

```kotlin
fun createTrip(name: String) {
    viewModelScope.launch {
        createTripUseCase(name)
            .onSuccess { trip -> _uiState.update { it.copy(navigateToTripId = trip.id) } }
            .onFailure { _uiState.update { it.copy(errorMessage = R.string.error_create_trip) } }
    }
}
```

#### UiState

```kotlin
data class TripListUiState(
    val trips: List<Trip> = emptyList(),
    val isCreateDialogVisible: Boolean = false,
    val navigateToTripId: Long? = null,   // consumed once navigation occurs
    val errorMessage: Int? = null          // string res id; consumed after shown
)
```

### Non-Functional Requirements

| Area | Requirement |
|---|---|
| **Performance** | Room insert runs on `Dispatchers.IO` (injected via DI); never block the main thread. |
| **Security** | No sensitive data in the trip name. No logging of user input. |
| **Offline-first** | All creation happens locally; no network call required. |
| **Future compatibility** | `TripRepository` interface must remain accessible from the domain layer so Android Auto (Car App Library) can consume it without touching the data layer directly. |
| **Configuration changes** | `isCreateDialogVisible` and text input state survive rotation via `rememberSaveable` or ViewModel state. |
